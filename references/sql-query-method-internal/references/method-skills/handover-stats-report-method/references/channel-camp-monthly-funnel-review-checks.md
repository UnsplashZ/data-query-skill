# 封板营期月度漏斗结论审阅检查

适用场景：用户给出飞书分析文档和「声乐/前端/封板营期/月度漏斗/按 market_belong」Excel，要求判断文档结论是否偏颇。

## 读取与对齐

- Excel 通常包含 `月汇总` 和 `月x渠道汇总` 两张表；先确认 sheet、列、月份范围、行数。
- 若飞书文档 API 不在评论上下文不可用，可用本机 Chrome 登录态抓取 DOM/截图；不要把远程 browser 的登录页当成文档不可读。
- 文档和 Excel 对照时，优先抽取文档中的：总盘结论、渠道结构判断、加微/D1/转化/ROI判断、分渠道经营建议。

## 关键审阅点

### 1. 识别“转化率”的真实含义

在该类报表中，`转化率` 可能沿旧 SQL 口径为：

```text
GMV / 2980 / leads
```

它不是严格的 `成交人数 / leads`。因此审阅文档时：

- 可以说“标准化 GMV 产出率下降”；
- 不要直接把它等同为“成交人数转化率下降”；
- 若文档进一步归因为“销售追单效率下降”，应标注为合理推断但证据不足，需要成交人数、支付时点、课后跟进、退款等数据验证。

### 2. 课下转化率是推算项

常见推算：

```text
课下转化率 = 总转化率 - 课中转化率
```

这适合做方向判断，但仍继承 `总转化率` 的 GMV 标准化口径，不应过度解释为纯人数或纯销售动作。

### 3. 拆结构影响与渠道内效率影响

当文档把总盘变化归因于渠道结构时，做一个 shift-share 检查：

```text
总变化 = Σ(new_share * new_rate) - Σ(old_share * old_rate)
结构影响 = Σ((new_share - old_share) * old_rate)
渠道内效率影响 = Σ(new_share * (new_rate - old_rate))
```

经验判断：

- D1 到课率下降经常会被私域/APP占比提升显著影响；
- 转化率/课中转化率下降未必主要由结构解释，可能更多来自渠道内效率下降；
- 因此不要把“D1结构变化解释”外推到“转化下降也主要由结构解释”。

### 4. 私域/APP不要只用“天然属性”解释

私域/APP通常低加微、低D1、低大盘课转化，应该单独建经营漏斗。但审阅时还要看其自身环比：

- leads 占比是否继续上升；
- D1、转化率、课中转化率是否自身也在下降；
- 如果占比稳定但效率继续下滑，不能继续主要归因为渠道结构。

### 5. ROI 要拆总盘和有成本渠道

总盘 ROI 可能被无直接成本渠道 GMV 托底。审阅时至少补看：

```text
有成本渠道 ROI = sum(GMV where leads成本 > 0) / sum(leads成本 where leads成本 > 0)
```

也可按经营大类看：BD1、BD2、直播、工作室、私域/APP、扶光、其他。

- 若总盘 ROI 仍大于 1，但有成本渠道 ROI 低于 1，文档应明确投放健康度风险。
- 对“值得加码”的渠道，必须拆到子渠道；不要只看大类 ROI。

### 6. 月汇总与渠道汇总一致性

校验：

- `月汇总.leads` 是否等于 `月x渠道汇总.leads` 合计；
- `月汇总.leads成本` 是否等于渠道合计；
- `月汇总.GMV` / `课中GMV` 是否等于渠道合计。

若 leads、成本能对上但 GMV 有 3%-5% 左右差异，常见原因是部分订单未映射到 `market_belong` 或渠道侧 join 丢失 GMV。此时：

- 总盘趋势仍可用；
- 分渠道 ROI/GMV 需要标注“以当前 market_belong 映射覆盖为准”；
- 不要把小比例未映射差异包装成精确渠道结论。

## 建议输出结构

1. 先判断文档主结论是否成立。
2. 再列出“成立的部分”和“偏颇/需加限定的部分”。
3. 对每个偏颇点给出 Excel 证据，如月度表、渠道大类聚合、shift-share 拆解。
4. 最后给出更稳妥的改写建议。

用户偏好：中文简洁直接，正式专业；不要只复述文档，要指出哪些结论证据足、哪些只是推断。
