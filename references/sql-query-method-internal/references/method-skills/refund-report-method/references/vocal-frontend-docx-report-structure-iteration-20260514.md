# 声乐前端 Docx 退费报告结构迭代（2026-05-14）

## 适用场景

用于已有 Feishu Docx 退费报告的数据和结论基本正确，但用户指出报告结构不符合业务分析逻辑，需要重组板块、补充 cohort 时效、提升专业表达。

## 本次确认的结构规则

1. 不要把「渠道维度」「交接状态解释边界」「问题-原因-建议」做成孤立板块。
2. 渠道、投手、交接状态都应进入对应 cohort 的诊断段落：
   - 3月M0：原因增量、渠道来源、文本信号、处理建议。
   - 3月M1：次月释放、退费间隔、原因结构、来源结构、处理建议。
   - 4月M0：参考3月M0，同样写问题定位、原因结构、时效结构、来源结构、处理建议。
3. 交接状态统一放在「数据与限制」中，作为风险分层变量，不作为因果结论依据。
4. 处理建议要嵌入各个分析板块，而不是最后集中列一个通用建议板块。

## 3月M1 退费间隔补充口径

当报告解释 `3月M1：次月释放退费` 时，需要补充 `支付到退费天数` 分层：

- `≤30天`
- `31-45天`
- `46-60天`
- `60天以上`

输出字段建议：

- 退费间隔分层
- 订单数
- 退费金额
- 金额占比
- 订单占比
- 中位天数

报告中应明确回答：

- 支付超过30天后退费多少单、多少金额、金额占比、订单占比。
- M1 不是 4月新增支付问题，但也不能只写成泛泛的「跨月释放」；要说明≤30天与31-60天各自的规模。

本次读数示例（声乐前端 2026-03 M1）：

| 退费间隔 | 订单数 | 退费金额 | 金额占比 | 订单占比 |
|---|---:|---:|---:|---:|
| ≤30天 | 60 | 73214.0 | 50.24% | 55.56% |
| 31-45天 | 31 | 47948.4 | 32.90% | 28.70% |
| 46-60天 | 18 | 24576.6 | 16.86% | 16.67% |
| 60天以上 | 0 | 0.0 | 0.00% | 0.00% |

支付超过30天后退费：48单，72525.0 元，金额占比 49.76%，订单占比 44.44%。

## Feishu Docx 自动化实现要点

1. 在 Excel 导出中新增 `3月M1退费间隔` sheet，避免报告中的时效判断不可核验。
2. 发布脚本校验不只检查新标题存在，也要检查被要求删除的旧独立标题不存在。
3. 发布后读回 `raw_content`，验证：
   - 新诊断标题存在。
   - 旧标题不存在。
   - `3月M1退费间隔` 和 `支付超过30天后退费` 等关键内容存在。
   - 图片 token 数量匹配。

## 语言要求

保持专业分析报告逻辑：

- 用「问题定位 / 原因结构 / 时效结构 / 来源结构 / 处理建议」。
- 渠道和投手描述为「用于定位复盘对象和处理优先级，不作为因果证据」。
- 避免口语化和强因果，例如：猫腻、确实异常、仍不低、只盯、一起看、造成、事故、系统性风险。
