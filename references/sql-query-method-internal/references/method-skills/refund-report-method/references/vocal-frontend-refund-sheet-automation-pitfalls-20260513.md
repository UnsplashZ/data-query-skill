# 声乐前端退费飞书表格自动化复盘要点（2026-05-13）

适用场景：把声乐前端某月退费、cohort、原因归类、交接状态分析写入现有 Wiki-backed 飞书表格。

## 本次验证过的输出结构

- `报告正文`
- `口径说明`
- `核心指标`
- `月度退费率`
- `cohort退费率`
- `cohort退费明细`
- `交接覆盖`
- `交接状态摘要`
- `交接状态月度`
- `原因归类摘要`
- `退费原因明细`
- `结构维度摘要`
- `结构维度月度`
- `4月退费明细Top1000`

## 关键口径

主范围：
- `cci3_name = '声乐'`
- `new_front_end_name LIKE '%前端%'`
- `main_first_level = '课程'`
- `pay_type_name IN ('全款','尾款')`
- `pay_status_name = '支付成功'`
- `pay_amount > 0`

核心指标：
- 同月 cohort 金额退费率 = 支付月订单在支付当月内发生的退费金额 / 支付月 GMV。
- cohort 矩阵保留 M0/M1/M2/M3/M4/M5；未成熟月份在解释中明确标注，不强行当最终退费率。
- 交接状态要拆已入群/入班、有交接未入群、无交接记录，避免整体均值掩盖无交接风险池。

## 技术坑

1. SQL 层建议继续使用 ASCII alias，再由 pandas rename 成中文列名。
2. 但 rename 后，后续 pandas 逻辑必须统一使用中文列名；尤其是 `groupby(...).agg(...)` 的 named aggregation：
   - 推荐：`agg(退费订单数=('退费订单数','sum'), 退费金额=('退费金额','sum'))`
   - 避免：`agg(refund_orders=(...), refund_amount=(...))` 后又引用 `退费金额`
3. `make_summaries()` 这类汇总函数中不要混用 `gmv` / `refund_amount` / `order_cnt` / `median_pay_to_refund_days` 与 `GMV` / `退费金额` / `订单数` / `支付到退费中位天数`。
4. 写飞书表格后必须 API 读回验证：至少读 `报告正文`、`核心指标`、`月度退费率`、`cohort退费率`、`4月退费明细Top1000` 的 A1:C5，确认标题和首行数据存在。

## 报告判断模板

- 不写“交接项目导致退费率变化”。
- 推荐写法：已入群/入班组内部退费率下降，方向上支持交接动作降低已承接客户风险；但整体退费率仍受无交接记录组、渠道/价格带/班型结构，以及历史 cohort 滞后退费释放影响。
- 若无交接记录组退费率明显上升，单列为风险池。
