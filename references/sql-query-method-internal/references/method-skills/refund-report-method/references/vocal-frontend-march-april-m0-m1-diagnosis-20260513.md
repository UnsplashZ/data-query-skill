# 声乐前端 3月M0/M1 与 4月M0 退费诊断补充

## 触发场景
用户要求不要只解释“4月读数受历史滞后退费影响”，而是进一步回答：
- 3月当月 M0 为什么已经很高；
- 3月 M1 是否继续释放、释放什么原因；
- 4月 M0 的退费结构、来源、原因是什么，并据此设计解决方案。

## 方法调整
在常规 cohort 报告外，新增三组重点 cohort：
- `2026-03_M0`：3月支付、3月退费，定位当月即时退费问题；
- `2026-03_M1`：3月支付、4月退费，定位次月释放问题；
- `2026-04_M0`：4月支付、4月退费，定位4月当月问题。

核心 SQL 逻辑：
- 支付 cohort：`formatDateTime(toStartOfMonth(f.pay_time),'%Y-%m')`；
- 滞后月：`dateDiff('month', toStartOfMonth(f.pay_time), toStartOfMonth(r.refund_time))`；
- 分子用真实退款明细表 `tock_dwd_order_refund_df.refund_time/refund_amount`，不是只用支付流表聚合字段；
- 分母用同支付月 `dwd_order_flow_df.pay_amount` 的 cohort GMV；
- 维度同时拆：`reason_cat/raw refund_reason/handover_status/handover_delay_bucket/price_band/market_belong/channel_emp/camp_sku/pay_to_refund_bucket`。

## 报告写法
这类报告要显式区分两类问题：
1. **当月即时退费问题**：M0 已经高，不能简单归因为“交接周期久导致次月退”。重点看 D0-D7/D8-D14、支付后预期管理、销售承诺边界、开课响应、客观约束是否前置识别。
2. **滞后释放问题**：M1/M2 继续释放，说明前月订单风险没有在当月完全暴露；用于解释4月发生退费的历史来源，但不能覆盖 M0 诊断。

推荐章节：
- 核心结论：同时列 3月M0、3月M1、4月M0 的金额和退费率；
- 3月M0/M1重点诊断：Top原因 + 支付到退费天数 + 渠道/投手来源；
- 4月M0问题定位：原因结构、来源结构、时效结构；
- 解决方案：首周预警、来源复盘、原因处理、交接SLA。

## 解决方案模板
- 首周预警：D0-D7早退订单每日监控，优先处理支付后未入群/未加微/未开课客户；
- 来源复盘：对 M0 退费额Top渠道、投手、营期SKU复盘销售承诺、价格权益说明、开课等待时间、班型匹配；
- 原因处理：健康/家庭/时间冲突做前置筛选与延期/转班选项；产品与服务匹配类做试听预期、课程难度和老师衔接话术；支付/价格类规范分期、权益和退费规则；
- 交接SLA：D0建联，D1入群/开课安排确认，D3前二次触达；无交接记录池单独追踪退费率。

## 注意事项
- 不要把3月M0高解释成“历史滞后退费”；M0本身就是当月即时退费。
- `reason_cat` 是规则归类，必须保留 raw reason 和 detail 方便业务复核。
- `分组内退费率` 只有能从支付订单表得到同维度分母时才展示；原因归类、支付到退费天数这类只存在于退款侧的维度不要伪造分母。
- 文案避免强因果，只写“定位为优先排查方向/经营假设”。
