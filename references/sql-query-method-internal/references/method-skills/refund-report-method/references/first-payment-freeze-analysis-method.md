# 首款订单 × 休学冻课诊断方法

适用场景：用户提供飞书表或明细，询问“冻课数据、仅首款数据怎么解析”。

## 核心口径

先明确不要把三类问题混在一起：
1. 首款订单大盘：课程｜首款支付成功｜未被尾款关联。
2. 首款中命中休学冻课：用流水号与冻课/休学明细交叉；通常筛 `类型=休学` 且 `是否匹配课程支付订单=1`。
3. 未分配班级：这是承接/归属问题，不能直接并入“首款冻课”结论。

## 推荐读取方式

如果数据在飞书 Wiki Sheet：
1. 用 `wiki/v2/spaces/get_node` 将 wiki token 解析为 sheet `obj_token`。
2. 用 `sheets/query` 获取 tab 列表与 grid size。
3. 汇总页可直接读全量；大明细页若超过 10MB，按 5000 行 chunk 读取，最后本地合并并 trim 空行。
4. 同一工作簿里常见有：首款订单汇总、首款订单明细、冻课明细。正式判断以明细交叉验证，不只看汇总单元格。

## 分析顺序

1. 先算首款中冻课占比：
   - 首款订单数
   - 首款且冻课订单数
   - 占比
   - 支付金额 / 退款金额 / 退款率
   低占比时，不要夸大为首款大盘问题，应转为局部定位。

2. 月度趋势定位异常：
   - 支付月份 × 首款订单数
   - 支付月份 × 首款冻课订单数
   - 首款冻课率
   先找异常月份，再展开，不要直接全量平均归因。

3. 拆 SKU：
   - 先看 SKU 贡献度；若声乐占主导，先声乐、再用钢琴/朗诵等作对照。
   - 输出订单数、支付金额、退款金额、退款率、学习行为率。

4. 拆前后端与学习行为：
   - 大前端且无学习行为：更偏支付后未真正进入学习/服务，重点看入班、触达、开课前承接。
   - 大后端且有学习行为：更偏学习中断/服务期暂停，重点看课程体验、时间冲突、学管处理、复课跟进。

5. 拆金额分桶：
   - 判断问题是否集中在低价体验课，还是中高客单首款。
   - 对用户的交接口径，价格分桶如涉及 1880/2980 时遵循已有规则：大于 2381 归 2980，小于等于 2381 归 1880。

6. 生命周期：
   用支付时间、首次休学时间、首次冻课时间计算：
   - 支付后 0–3 天
   - 4–7 天
   - 8–14 天
   - 15–30 天
   - 31–60 天
   - 60 天以上

   解读：
   - 0–7 天：售前承诺、开课安排、支付后反悔、时间冲突。
   - 8–30 天：入班/首轮学习体验。
   - 30 天以上：长期坚持、服务维护、客观原因。

7. 定位对象集中度：
   - 营期名称
   - 交接班级名称
   - 交接学管
   - 成交人
   订单量小的时候只做“集中发生对象定位”，不要对个人定责。

8. 冻课后退款分层：
   - 冻课未退款：维护池/复课机会。
   - 冻课已退款：损失池。
   - 大后端 + 有学习行为 + 已退款：优先复盘池。

## 报告结构

1. 结论页：是否大盘问题、异常集中点、前后端机制差异。
2. 趋势页：月度首款冻课率，标出异常月份。
3. 结构页：SKU、前后端、金额分桶、学习行为。
4. 生命周期页：支付到首次冻课、冻课到退款。
5. 定位页：异常月份/SKU 下的营期、班级、学管、成交人。
6. 动作建议：
   - 大前端：支付后 7 天内触达、入班、开课提醒、未学习预警。
   - 大后端：有学习行为后冻课原因标注、复课跟进、冻课转退款预警。

## Pitfalls

- 汇总页格式可能有异常，例如金额被套成百分比。正式结论应优先用明细交叉验证。
- `是否有学习行为` 可能是中文枚举（有学习行为/无学习行为），不要只按 1/0 判断。
- 首款冻课样本量可能很小；此时应定位异常月份/班级/营期，不宜过度泛化。
- 不要把“未分配班级”的高退费率直接解释为“首款冻课”问题，它是另一条承接归属链路。
