# 钢琴 cohort 退费原因分析（2026-05-20）

## 触发场景
用户询问某 SKU 最近几个月的 cohort 退费率，并继续要求分析退费原因。这个参考记录了钢琴案例中已经验证过的 ClickHouse 表、口径和输出结构。

## 数据源与口径

- 支付 cohort 分母：`dwd_order_flow_df`
  - `pay_status_name = '支付成功'`
  - `cci3_name = '钢琴'`
  - `pay_time >= toDateTime('2026-01-01 00:00:00')`
  - `pay_time < toDateTime('2027-01-01 00:00:00')`
  - 分母：支付月 `sum(pay_amount)`
- 退费明细/原因：`tock_dwd_order_refund_df`
  - `cci3_name = '钢琴'`
  - `refund_amount > 0`
  - `refund_reason`, `refund_reason_detail`
  - cohort 按支付月归因：`toStartOfMonth(pay_time)`
  - lag：`dateDiff('month', toStartOfMonth(pay_time), toStartOfMonth(refund_time))`

注意：总 GMV 不要用 `main_goods_sku` 收窄。此前用户明确纠正：总 GMV 用 `cci3_name=声乐/钢琴`；只有“880+课程某商品订单 GMV”这类筛选子集才用 `main_goods_sku=声乐/钢琴 AND main_first_level='课程' AND total_original_price>=880`。

## 已验证输出形态

Excel 输出建议包含：

1. `00_口径说明`
2. `01_整体cohort`
3. `02_前后端cohort`
4. 若用户追问原因，再生成：
   - `01_3月4月大类对比`
   - `02_原因大类`
   - `03_原因主题`
   - `04_原始原因`
   - `05_退费明细`

用户-facing 表头用中文，金额单元格保持数值格式，百分比列用百分比格式。

## 原因分类规则（规则归类，不是质检归因）

- `学员客观约束`：家人、身体、时间
- `支付/价格机制`：经济、尾款、定金、错误付款、重复付款、价格、实付、付款、差价、降价、钱
- `产品与服务匹配`：课程、主讲、辅导、老师、无法满足、竞品、不喜欢、想象、质量、学习机、琴、产品
- `其他长尾`：其余

主题细分可用：

- 家人反对
- 时间冲突
- 身体健康
- 经济/尾款压力
- 定金认知偏差
- 误付/重复付
- 主讲老师不匹配
- 辅导老师不匹配
- 课程内容不匹配
- 特殊诉求无法满足
- 转向竞品
- 其他

## 钢琴案例中的关键读数

整体 cohort：

- 2026-03：支付 GMV 11,686,993.57；M0 1.63%；M0_M1 2.99%；当前累计 3.20%。
- 2026-04：支付 GMV 8,654,453.99；M0 3.45%；M0_M1 4.29%；当前累计 4.29%。
- 2026-05：支付 GMV 5,314,525.70；当前累计 1.99%，月中未成熟，不宜直接和完整月份比较。

前后端拆解：

- 2026-04 大前端：M0 4.16%；M0_M1 5.67%。
- 2026-04 大后端：M0 2.78%；M0_M1 2.98%。

原因分析：

- 钢琴 2026 累计金额最高的原始原因：家人原因、不让学习；时间原因；身体原因。
- 2026-04 大前端仍以家人/身体/时间为主，三项合计约 13.14 万，占退费额 54.90%；产品与服务匹配类较 3 月下降。
- 2026-04 大后端异常更集中于时间原因：时间冲突约 7.89 万，占 59.66%；学员客观约束合计约 10.97 万，占 82.88%。
- 支付/价格机制类订单数不少，但金额占比通常不是最高；适合作为支付页、定金/尾款说明和规划师话术的治理项。

## 报告解释边界

- `refund_reason` 是业务/审批文本，可能存在多原因并列；规则分类用于看结构，不等同于严格质检归因。
- 当前月和次月 cohort 不成熟时，必须写明成熟度限制。
- 对“原因导致退费率上升”的表达要谨慎，优先写“结构上主要由……贡献”“提示需要抽样复盘……”。
