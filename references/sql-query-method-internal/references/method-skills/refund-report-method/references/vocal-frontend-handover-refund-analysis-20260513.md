# 声乐前端 4月交接项目与退费分析（2026-05-13 方法摘录）

适用场景：用户要评估某月前端开课/交接优化对退费的影响，尤其是“当月退费率较过去半年变化，可能与历史交接周期长、次月/次次月退费释放有关”。

## 推荐口径

主范围示例：
- `cci3_name = '声乐'`
- `new_front_end_name LIKE '%前端%'`，覆盖 `大前端` 与 `大前端-华彩乐园`
- `main_first_level = '课程'`
- `pay_type_name IN ('全款','尾款')`
- `pay_status_name = '支付成功'`
- `pay_amount > 0`

主指标：
- 同月 cohort 金额退费率 = 支付月订单在支付当月内发生的退费金额 / 支付月 GMV。
- cohort 滞后矩阵：M0=支付当月退费，M1=次月退费，M2=次次月退费，继续到 M5/M6 视需求。
- 4月项目评估必须同时看：
  1. 4月支付 cohort 的 M0 退费；
  2. 4月发生退费来自哪些支付月；
  3. 2月/3月历史 cohort 在4月的滞后退费释放。

## 交接影响判断框架

不要直接把“4月总体退费率上升/下降”归因于交接项目。

同时拆三层：
1. 大盘同月退费率：4月 vs 3月、前3月均值、前6月均值/加权均值。
2. 交接过程覆盖：交接、入群/入班、加微，以及支付到交接中位天数。
3. 同交接状态内退费率：已入群/入班、有交接未入群、无交接记录。

解释规则：
- 若已入群/入班组内部退费率下降，但总体退费率不降，优先判断为“已承接客户风险下降，但被未承接客户、渠道/价格带/班型结构或历史 cohort 滞后退费抵消”。
- 若无交接记录组退费率升高，需要单列为风险池，不要用整体均值掩盖。
- 交接覆盖率本身不等于效果；业务报告中应强调结果指标与 cohort 追踪，而不是把“有没有交接记录”当成可见 KPI。

## 推荐输出表

- `报告正文`：结论、交接判断、cohort解释、退款原因、结构风险、建议动作、限制说明。
- `口径说明`：明确 SQL 谓词、数据源、因果限制。
- `核心指标`：GMV、同月退费额、同月金额退费率、过去半年基准、入群/入班率、加微率、支付到退费中位天数。
- `月度退费率`：最近 6~7 个支付月。
- `cohort退费率`：M0/M1/M2/M3/M4/M5 金额与累计退费率。
- `cohort退费明细`：支付月 × 退费月 × lag。
- `交接覆盖`、`交接状态摘要/月度`。
- `原因归类摘要/明细`。
- `结构维度摘要/月度`：渠道、投手、价格带、营期/班型、前后端细分。
- `4月退费明细Top1000`：保留原始原因与补充说明。

## 报告语言

- 明确区分事实、推断、限制：这是观察性分析，不是强因果证明。
- 避免“4月交接优化导致退费率变化”这类强归因。
- 推荐表达：“与交接动作方向一致 / 支持该假设 / 仍需用后续 M1/M2 cohort 验证”。
- 对历史滞后退费使用“释放”或“承接历史 cohort 风险”，不要混成 4月支付订单本身的问题。

## 技术注意

- 原因明细应优先来自退款明细表（例如 `tock_dwd_order_refund_df`）以获取 `refund_reason` / `refund_reason_detail`。
- 交接维度关联 `dwd_order_handover_df` 前先按 `flow_no` 去重，避免一单多交接记录放大订单行数。
- ClickHouse 输出给 pandas/飞书时，SQL alias 推荐使用 ASCII，落地 DataFrame 后再 rename 为中文列名；避免中文 alias 在服务端异常时触发 `clickhouse_driver` 异常信息解码问题，导致真实 SQL 错误不可见。
